## この手順書の前提と読み方

これまではさくらの**レンタルサーバー**で運用してきました。レンタルサーバーは、Webサーバー・PHP・MySQL・メール送信・SSL証明書などがあらかじめ用意された状態で借りるものでした。

今回移行する**さくらのクラウド（IaaS）**は、借りるのが「空のLinuxサーバー1台」です。OSとログインユーザーは用意されますが、Webサーバーもデータベースも何も入っていません。それらを自分でインストール・設定し、動き続けるよう管理する——これが本手順書の作業内容です。

作業は大きく4段階に分かれます。

1. 土台づくり（サーバーを作り、ログインして下ごしらえ）
2. 道具のインストール（アプリを動かすためのミドルウェア）
3. アプリを載せて動く形にする（デプロイ）
4. 公開と本番特有の設定（ドメイン・SSL・キューワーカー・メール等）

引き継ぎ資料にあった5つの本番化タスク（キューワーカー常駐・CORS本番化・mediaパスのLinux化・mov変換タイムアウト解消・メール実アドレス化）は、いずれも第4段階に位置づけられます。第1〜第3段階は今回新しく組む部分です。

### 注意事項

- 各コマンドを実行する前に、その作業が何のためか（コメント）を確認する。
- 認証情報（DBパスワード・ストレージのシークレット・メールのパスワード等）は本番 `.env` に書くが、**リポジトリにコミットしない**。`.env` はサーバー上でのみ管理する。
- ディストリビューションは Ubuntu 24.04 LTS を前提とする。Webサーバーは nginx + PHP-FPM を前提とする。DB は同一サーバーに同居させる方針。

- **運用前提**：この環境は使い捨てのデモ環境ではなく、お客様へのデモの後、近日中にそのまま本番運用へ移行する。したがって最初から本番として作り込み、常時稼働（月額固定）を前提とする。デモ用に簡易環境を別途作る運用はとらない。

## 料金・課金の考え方（着手前に把握しておく）

レンタルサーバーは「月額固定で借りる」ものだったが、さくらのクラウド（IaaS）は**リソースを作った時点から課金が始まる従量課金制**である。ここを理解しておかないと、想定外の請求につながる。

### いつ課金が始まるか

- **サーバー**：仮想サーバーの作成完了・**起動した時点から課金開始**、**停止すると課金停止**。「＋追加」で作って起動した瞬間から料金が発生する。
- **ディスク**：**作成した時点から、削除するまで課金**が続く。サーバーを停止しても、**ディスクを削除しない限りディスク分の料金は発生し続ける**点に注意（レンタルサーバーには無かった感覚）。
- リソースを何も追加していない状態（会員登録だけ）では課金されない。

### 料金体系のしくみ

- 利用時間に応じて**自動的に一番安い料金が適用**される。1か月の稼働時間が10時間以内なら時間料金、20日までは日額料金、それ以降は月額料金に自動で切り替わる。本番サーバーは24時間稼働させるため、実質「月額固定」になる。
- **データ転送量・リクエスト数による課金がない**。写真・動画を配信する本システムでも、アクセス増で通信料が跳ねる心配がない（AWS等との大きな違い）。円建てで為替変動リスクもない。

### おおよその金額感

- 最小構成の目安：サーバー（メモリ1GB）月額 約1,540円＋ディスク（40GB）月額 約880円＝**約2,420円から**。
- ただし本システムは Web・PHP・MySQL・キューワーカーを1台に同居させるため、メモリ1GBでは不足する。参考として **2vCPU／4GB／SSD100GB 構成で月額 約8,690円** という試算がある（時期・条件により変動）。
- 実際の金額は次段階で決める**スペックに直結する**。

### 課金前・課金後の安全策

- 実際に作る前に、公式の**料金シミュレーション**（計算ツール）で見積もれる。スペックを決める際に必ず利用する。
- 運用開始後は、コントロールパネルの利用料金管理から**料金アラート機能**を設定できる。請求が一定額を超えそうなときにメール等で事前通知を受けられる。設定しておくことを推奨。
- 使わないサーバーは停止すればサーバー料金は止まる（ただしディスクは残る）。開発・検証用に一時的に立てたサーバーは、用が済んだら停止・削除して無駄な課金を避ける。

※料金体系は改定されることがある。着手時点で公式の料金ページ・シミュレーションで最新値を確認すること。

## スペックは後から変えられる（スペック選びの前提）

「最初に完璧なスペックを当てる」必要はない。さくらのクラウドは後からスペックを変更できるが、**CPU・メモリとディスクで変えやすさが大きく異なる**。この違いが、初期スペック選びの方針を決める。

### CPU・メモリ：気軽に変えられる（両方向）

- 「プラン変更」機能で CPU コア数・メモリを変更できる。**増やす（スケールアップ）も減らす（スケールダウン）も両方可能**。
- 手順は **サーバーを停止 → コントロールパネルで対象サーバーを選び「詳細」→「プラン変更」→ 新プラン選択 → 起動**。OSやアプリの入れ直しは不要。
- 注意点：
  - **サーバーの停止が必要**（変更中はサービスが止まる。本番稼働後はアクセスの少ない時間帯に実施）。
  - 内部的には別スペックのサーバーにディスク・基本情報をコピーし元サーバーは自動削除される仕組みのため、**リソースIDが変わる**（通常のWeb運用では影響は小さいが、IDに依存した管理をしている場合は留意）。
  - ※参考：さくらの「VPS」はスケールダウン不可だが、今回使う「クラウド」は両方向いける。

### ディスク：増やせるが手間がかかる。縮小は不可

- 今使っているディスクを**その場で直接大きくすることはできない**。より大きいディスクを新規作成し、作成時に既存ディスクを指定してデータをコピー → 「パーティションサイズの拡張」で新容量まで広げる → サーバーの接続先を新ディスクに付け替える、という流れ。**データは保たれる**が工程が多く、コピーに時間がかかる（容量により数分〜数時間）。作業中はサーバーを停止する。
- **ハマりどころ**：大きいディスクにコピーしただけでは容量が増えない（元サイズ分しか使われない）。「パーティションサイズの拡張」の一手間が必須。パブリックアーカイブ由来のLinux（ext系ファイルシステム）なら補助機能が使える。
- **縮小はできない**（小さくするには手動データ移行が必要で非常に手間）。作りすぎると下げにくい。

### スペック選びの方針（この違いから導かれる結論）

- **CPU・メモリは控えめに始めてよい** → 後から両方向に調整できるので、まず実用最小限で立て、本番の負荷を見てから増減する。無駄な課金も抑えられる。
- **ディスクはやや余裕を持たせる** → 後で増やすのは手間、縮小は不可のため。ただし本システムは写真・動画の本体をオブジェクトストレージに逃がす設計で、サーバー本体のディスクにはOS・アプリ・DBしか乗らない。DBの増加を見込んでも極端に大きくする必要はなく、「ケチりすぎず妥当な余裕」程度でよい。

## デモ準備で発見・修正したバグ（記録フォームの Blade 構文エラー）

- 事象：「クライアント詳細 → トレーニング記録の新規登録」（`/training-records/create?client_id=1`）で 500 ParseError。開発・本番の両方で再現（コード側の問題）。
- 原因：`src/resources/views/training-records/_form.blade.php` の **JSコメント（`//`）内に生の `@once @push('styles')` が書かれていた**（563行・885行、コミット 5f746be の技術メモ追記時に混入）。Blade はJS/HTMLの文脈を認識せず `@` ディレクティブを全域から拾うため、`@once` が閉じられず（`@endonce` 無し）コンパイル済みPHPが壊れ、EOFで `expecting endif` エラーになった。
- 教訓：**Blade コメント `{{-- --}}` 内の `@` は安全だが、JSコメント `//` 内の `@` は Blade に拾われる**。ディレクティブ名をコメントに書くときは `@@` でエスケープする（例：`@@once`）。`@@` は最終出力でリテラル `@` 1つになるので可読性も保たれる。
- 修正：563行・885行の `@once @push` → `@@once @@push`（コミット 4b20b15、2行のみ）。開発→main→本番pull。**本番は pull 後に `php artisan view:clear`** で壊れたコンパイル済みビューを破棄（必須。放置すると古いキャッシュで同じエラーが出続ける）。
- 確認：本番でクライアント（佐藤太郎）ログイン → トレーナーが作成した記録＋紐づけメディア（動画・再生アイコン付き）がクライアント側で閲覧できることを確認。**デモの主要フロー（記録作成→メディア紐づけ→クライアント閲覧）が本番で完動**。
- 発見経緯：デモ用サンプルクライアントを開発で検証中に発見。開発先行のおかげでデモ前に捕捉できた。

## iPad（Safari）で一覧の行タップが最下行しか開かない不具合 → 修正済み

- 事象：クライアント一覧・トレーニング記録一覧で、iPad（Safari）だとどの行をタップしても「一番下の1件」の詳細が開く。**PC（Chrome/Edge）では正常**。
- 原因：行が `<tr class="position-relative">` + Bootstrap `stretched-link` 方式。**iPadOS Safari（WebKit）は `<tr>` に position:relative を効かせない**（テーブル行を positioned containing block として扱わない実装差）。そのため各行の stretched-link の `::after` オーバーレイが `<tr>` に収まらずビューポート全体に広がり、全行が重なってDOM順で最後の `<tr>` が全タップを吸収。→ 症状は入力方式（マウス/タッチ）ではなく**レンダリングエンジン差**（Blink vs WebKit）が本質。Bootstrap公式も「stretched-linkはテーブル行では信頼できない」と警告している既知問題。
- 修正：`clients/index.blade.php` と `training-records/index.blade.php` の行を、既存 `clients/show.blade.php` と同じ `<tr style="cursor:pointer;" onclick="location.href='...'">` 方式に統一（stretched-link と position-relative を除去）。コミット 9f56577、開発→main→本番pull→`view:clear`。CSS無変更（ビルド不要）。
- 教訓：Bootstrap `stretched-link` を `<tr>` に使うと iOS/iPadOS Safari で誤作動する。行全体クリックは `<tr onclick="location.href=...">` 方式が確実（マウス・タッチ・エンジン差に強い）。
- ※修正後、iPad Safari のキャッシュが残ると旧挙動が出ることがある（再読み込み or サイトデータ消去で解消）。

## 記録フォームの未保存確認がメディア追加/削除で出ない不具合 → 修正済み

- 事象：トレーニング記録の登録・編集画面で、未保存確認（`UnsavedChangesGuard`、「保存されていない変更があります。移動しますか？」）はテキスト項目では実装済み・動作するが、**後から追加したメディアの追加/削除/並び替えでは未保存確認が出ず**、変更が警告なく失われる。
- 原因（三重の漏れ、investigation判明）：`resources/js/unsaved-changes.js` の `UnsavedChangesGuard` は (1) hidden input を監視対象外にし (2) init時の固定リスト方式（動的追跡なし）で (3) メディア操作（`mediaSelection` の add/remove/move → render で hidden input を作り直す）が input/change を発火しない。メディアの永続化先が `<input type="hidden" name="media_record_ids[]">` のみのため、既存の dirty 検知に一切引っかからなかった。
- 修正（既存の仕組みに乗せる最小変更、コミット be77e5f）：
  - `unsaved-changes.js` に公開API `markDirty() { this.hasChanges = true; }` を追加。
  - create/edit.blade.php でインスタンスを `window.unsavedChangesGuard` に露出。
  - `_form.blade.php` の `mediaSelection` の add/remove/move の3メソッドで `window.unsavedChangesGuard?.markDirty();` を呼ぶ（init には入れない＝初期表示が未保存扱いになる副作用を回避）。
- **重要な教訓（JS変更時のビルド必須）**：今回は Blade だけでなく `resources/js/unsaved-changes.js`（Viteバンドル対象、app.js から import）を変更したため、**開発でも本番でも `npm run build` が必須**。ビルドを忘れると「Blade側は新しい（`window.unsavedChangesGuard=...`）がバンドルされた `window.UnsavedChangesGuard` は古く markDirty が無い」ちぐはぐ状態になり、`Uncaught TypeError: window.unsavedChangesGuard?.markDirty is not a function` でメディア追加自体が停止する（`?.` は null/undefined は救うが「メソッドが無い」は救えない）。開発で先にビルド漏れに気づけたため本番事故を回避。
- 本番反映手順（JS変更を含む場合）：`git pull` → `cd src` → **`npm run build`** → `php artisan view:clear`。※Bladeのみの変更ならビルド不要だが、`resources/js/` 配下を触ったら必ずビルド。
- 確認：開発・本番とも、メディア追加が正常動作し、メディア追加/削除後のキャンセルで未保存確認が出ることを確認。
