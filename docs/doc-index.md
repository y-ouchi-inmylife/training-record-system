# 設計書索引

トレーニング記録管理システムの設計書を管理する。

---

## 1. 設計書の体系

本プロジェクトは Claude Code を主な実装手段として開発する。
従来の SI 開発における「基本設計書／詳細設計書」という階層的な分類はせず、
**役割** で 3区分する。

### 3区分

- **要件文書**: 何を作るか（What）を定義する。お客様視点。
- **仕様文書**: どう作るか（How）を定義する。開発者視点。
- **知識文書**: 全員が前提とすべき共通知識。

## 3. 文書間の参照ルール

### 要件定義書 → 仕様文書 の参照
要件定義書から仕様文書への参照は **基本的にしない**。
要件定義書は独立した正典として完結する（要件定義書作成時点では仕様文書はまだ存在しない）。

### 仕様文書 → 要件定義書 の参照
- 機能（6-X-X）への参照: **OK**
- 画面（7-X-X）への参照: **OK**
- バッチ（8-X-X）への参照: **OK**
- データ項目（9-X-X）への参照: **OK**

### 仕様文書間の参照
- 画面 ↔ バッチ ↔ API ↔ DB は相互参照 **OK**
- 例: API設計書から「対応する画面: screen-design.md S-XXXX」

### 用語集
- どの文書からも参照される
- 用語集は他文書を参照しない（独立）

### 実装コード → 設計書 の参照（コメント・PHPDoc）
- 安定したID（画面ID S-XXXX、データ項目ID D-XXXX、バッチID B-XXXX）への参照: **OK**
- 章番号・節番号（例: 「3.3」「4.7.12」）への参照: **しない**
  - 設計書の改訂で番号がずれ、参照が陳腐化して混乱の元になるため
  - 章・節を指したい場合は、番号ではなく見出し名や安定IDで参照する

## 4. AI（Claude Code）向け指示

各文書を Claude Code に読ませる際の指示:

#### requirements.md（要件定義書）
- **何の真実か**: 製品が何を実現するか（要件）
- **AIへの指示**: この文書は要件の正典である。仕様文書と矛盾する場合、この文書が優先される。
- **更新タイミング**: 要件変更時。改訂履歴に記録する。
- **注意事項**: 本文書はお客様向けの業務要件文書であると同時に、開発者・AIが仕様文書を作成するための起点でもある。業務要件はお客様視点で書く。設計思想（章構成と役割など）は開発・実装の起点として必要であれば含めて良い。お客様に見せる際は、設計思想など開発・実装向けの記述を削除して使う。

#### architecture.md（アーキテクチャ設計書）
- **何の真実か**: システム全体の技術構造
- **AIへの指示**: この文書は技術選定の判断基準である。新しい実装の方針はここに従う。
- **更新タイミング**: 技術スタック変更時
- **注意事項**: 全体の構造を俯瞰する文書。各論の詳細は他の仕様文書に委ねる。

#### screen-design.md（画面設計書）
- **何の真実か**: 各画面の動作仕様
- **AIへの指示**: この文書は画面実装時の真実である。
- **更新タイミング**: 画面仕様変更時
- **注意事項**: ワイヤーフレーム・画面遷移・操作仕様を記述する。

#### batch-design.md（バッチ設計書）
- **何の真実か**: artisan コマンドの実装仕様と定時実行設定
- **AIへの指示**: この文書はバッチ実装・スケジュール変更時の真実である。
- **更新タイミング**: バッチ追加・変更時、定時実行スケジュール変更時
- **注意事項**: 業務ルール（削除期間、ロック期間、実行時刻等）は要件定義書側に置き、こちらには技術仕様のみ記述する。

#### db-schema.md（DB設計書）
- **何の真実か**: データベースのテーブル構造
- **AIへの指示**: この文書はマイグレーション作成時の真実である。
- **更新タイミング**: テーブル変更時
- **注意事項**: 開発者向けの詳細仕様を記述する。

#### api-design.md（API設計書）
- **何の真実か**: API のインターフェース仕様
- **AIへの指示**: この文書は API 実装・変更時の真実である。
- **更新タイミング**: API 変更時
- **注意事項**: エンドポイント・リクエスト・レスポンスを記述する。

#### glossary.md（用語集）
- **何の真実か**: 用語の定義
- **AIへの指示**: この文書は用語の意味が不明な時に参照する。
- **更新タイミング**: 新しい用語が登場した時
- **注意事項**: 業務用語を中心に記載する。技術用語は最小限。

### 文書を更新する際の原則

1. **現状を書き直す**: 追記ではなく上書き。古い記述は削除する。
2. **改訂履歴は git に任せる**: 要件定義書を除き、文書内に履歴を書かない。
3. **Why を残す**: 「なぜそうしたか」の判断根拠を残す。
4. **How はコードに任せる**: 実装詳細は文書に書かない。

## 5. 推奨作成順序

新規プロジェクトで設計書を作る場合の順序:

1. **要件定義書** — 全体の土台。機能・画面・データ項目・バッチを確定させる
2. **アーキテクチャ設計書** — 技術方針の決定
3. **画面設計書 + バッチ設計書 + API設計書** — 並行作成
4. **DB設計書** — API とデータ確定後
5. **用語集** — プロジェクト全体を通じて随時更新

---

## 6. 設計書の対象外

以下は本プロジェクトの「設計書」ではなく、別カテゴリーで管理する:

- **テスト関連**: tests/manual-test-checklist.md, tests/bugs.md
- **運用関連**: startup-guide.md, sakura-deployment-guide.md
- **契約関連**: contracts/contract-template.md, contracts/contract-{機関名}.md
  - 機関ごとの契約条件（金額、開発期間、検収日、保守期間等）を記載
  - 要件定義書 14〜16章のフレームワークに基づき、具体内容を記載
  - 機密情報のため、共有範囲を限定
  - **運用ルール**:
    - `contract-template.md` のみ git 管理
    - 個別契約書（`contract-{機関名}.md`）は `.gitignore` で除外
    - 確定版は 1Password 等の暗号化ツールで保管
