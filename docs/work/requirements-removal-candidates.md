## 削除作業の手順

### 着実に進めるための原則

- 1カテゴリずつ確認 → 削除コミットを分ける
- 削除前に大内さんが「本当に削除でよい」と判断 → 確認状況を ⏳ → ✅ に更新
- 削除実行は Claude Code 等で大内さんが実施
- requirements.md を更新したら、その差分(commit)を記録

### 推奨する削除順序

最も影響の大きい・判断が明確なものから:

1. **A. 8章で言及済み**(4項目=9-1全体):最も明確。8章で代替できる
2. **C. オンプレ前提の古い記述**(14項目):実態と乖離しているので、残す方が害が大きい
3. **B. 他項目との完全重複**(5項目):重複先が明確
4. **D. architecture.md に既存記述あり**(11項目):移動先の整合性をarchitecture.md側で確認してから

この順序で進めると、9〜12章の本体がだいぶスッキリする。
