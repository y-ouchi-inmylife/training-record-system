## 判断履歴

### 2026-05-07：A案（要件定義書に掲載しない）で確定

論点6（サービスレベルの章配置と表現）の検討結果：

**結論**：要件定義書には掲載しない（旧9-2-4の削除のまま、復活させない）

**判断理由**：
1. 業務実態との整合
   - 個人事業主・中小機関中心のお客様で、SLA文書を求めるレベルではない
   - 14章「受入テストなし、本番稼働後のサポート重視」の方針と整合

2. 業者側のコントロール範囲外の要素が多い
   - さくら共有サーバーの稼働 → 業者がコントロール不可
   - 回線速度 → お客様側の環境次第
   - これらに数値約束を出すのはリスク

3. 既存の章で十分カバー
   - 9-2-3「データの保管期間と整理」で運用方針は触れている
   - 16章「保守」でサポート体制は明示

4. 書く場合のリスクが大きい
   - 「3秒以内を目標」が未達成だとクレームの元
   - 「24時間365日」が止まった場合の責任問題

**検討した代替案**：
- B案：16章保守に統合（定性化して残す）→ 紛争リスクは下がるが情報の重複感が出る
- C案：4-2 利用環境の補足として残す → 4章は「説明」の章なのでサービスレベルは性質が違う

**ファイルの位置づけ**：
- 本ファイルは git 管理外（docs/work/）
- 将来「サービスレベルを文書化したい」という議論が出た場合の検討履歴として保管
