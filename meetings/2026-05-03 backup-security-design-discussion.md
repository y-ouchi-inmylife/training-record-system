# 議事録：バックアップセキュリティ設計の議論

- 日時：2026年5月3日（午前・午後・お散歩後）
- 目的：要配慮個人情報を扱うシステムのバックアップ機能のセキュリティ設計

---

## 0. 議論の経緯（サマリ）

本議論は3つのフェーズで進行した：

1. **午前の議論**：論理削除+ゴミ箱という重い設計を組み立てた
2. **お散歩での思考**：業務実態に立ち返り、設計を本質的に見直した
3. **午後の議論**：シンプルで一貫性のある設計に再構築

最終的に、「対応すべき3つのリスクシナリオ」という軸で整理し直した。

---

## 1. 議論の背景

### 現状のバックアップ機能（要件定義書 6-11）

- **6-11-1 自動バックアップ**：毎日決まった時刻にバックアップ。デフォルト 03:00。直近7回分を保持
- **6-11-2 手動バックアップ**：任意のタイミングで手動実行
- **6-11-3 バックアップファイルの履歴**：一覧表示、ダウンロード、復元、削除が可能
- **アクセス権限**：管理者、システム管理者
- バックアップファイルは `.sql` ファイル（DBダンプ形式、平文）
- クライアント氏名・生年月日・住所・相談内容などの個人情報が平文で含まれる
- 画面からダウンロード可能、ダウンロード後の取り扱いは管理者の運用に委ねられる

### 当初の懸念

- 持ち出しリスク（ローカルPC紛失・盗難、退職時持ち出し）
- 誤送信リスク（メール添付、クラウドストレージ誤UP）
- マルウェア感染リスク
- 権限濫用リスク
- 保管リスク（サーバー上のバックアップファイルが侵入者に読まれる）

### 法令上の評価

- カウンセリング記録は **要配慮個人情報**（個人情報保護法 第2条第3項）
- 個人情報保護委員会のガイドラインで4種類の安全管理措置が要求
  - 組織的・人的・物理的・技術的安全管理措置
- 「外部に持ち出す場合の漏えい等防止」として暗号化・パスワード保護が明記
- 2022年4月施行の改正個人情報保護法で漏えい時の報告義務・本人通知義務が法定化
- 要配慮個人情報の漏えいは1件でも報告義務が発生

### 業界水準との比較

- 医療系（電子カルテ等）：3省2ガイドライン準拠、原則暗号化、ダウンロード機能を持たない設計が多い
- 心理臨床系SaaS：管理者にダウンロード機能を非公開、復元はサポート経由
- 一般SaaS：ダウンロード可能だが暗号化、または操作監査ログ完備

## 2. 議論の最終整理 - 3つのリスクシナリオ

お散歩を経て、以下3つのリスクシナリオに対応する形で整理し直した：

1. 操作ミスなどで削除したデータの復旧（クライアント、相談記録）
2. さくらサーバーのアカウント削除ミス
3. ローカルLAN環境でのサーバー破損

これらに対する設計を、それぞれ次の章で記述する。

---

## 3. 第1のシナリオ：操作ミスでのデータ復旧

### 確定事項

- **基本方針**
  - 物理削除のまま（現状維持）
  - 業務ルールガードで致命的事故を防ぐ
    - 相談記録があるクライアントは削除不可
    - 相談記録削除時、参加者も同時削除
  - 削除前の確認ダイアログで誤操作を予防

- **論理削除の扱い**
  - 標準機能としては実装しない
  - 将来「データを残したい」事業所が現れたら、その事業所向けに論理削除オプションを追加実装
  - 機関ごとの設定で切り替え可能にする可能性

- **クラウド版・LAN版の扱い**
  - どちらも物理削除で統一
  - 設計分岐なし

### この結論に至った経緯

#### 午前の議論（重い設計を組み立てた）

午前の議論では「論理削除+ゴミ箱+30日保持+物理削除バッチ」という重い設計を組み立てた：

| 項目 | 午前の結論 |
|---|---|
| 対象テーブル | clients、counseling_records、counseling_participants |
| 保持期間 | 30日 |
| 権限 | 管理カウンセラーのみ |
| UI | タブ切替、残日数表示 |
| 物理削除バッチ | 毎日03:00 |

#### お散歩での見直し

お散歩で業務実態に立ち返って整理した結果、以下の気づきが生まれた：

**気づき1: 削除実態がレア**
- 削除実態：稀（管理カウンセラーが年に数回）
- データ量：テキストのみで容量圧迫の可能性は低い
- 自動削除はやりすぎ、手動削除で十分

**気づき2: ゴミ箱の矛盾**
- 「完全削除」をクライアント一覧・相談記録一覧に配置するのが機能的に妥当
- しかしシステム管理者は業務データに触れない設計原則
- 一方で「削除できないゴミ箱」は管理カウンセラーが気持ち悪い

**気づき3: 業務的なリアリティ**
- 相談記録：内容の濃さが曖昧、間違って消してもカウンセラーの記憶で再登録できる
- クライアント：相談記録があれば削除できない（業務ルールガード）
- 重要なクライアントを誤削除することは構造的に起きない

**気づき4: 今受注している事業所の運用**
- クライアントや相談記録を「消さない」運用
- 削除されたクライアント = 間違って入力した人（誤入力の修正）
- 「誤削除」のリスク自体がそもそも杞憂に近い

**気づき5: 法令対応との戦略的相性**
- 「利用が5年ないクライアントのデータは消す」ルールが将来できる可能性
- もしそのルールができた場合、ゴミ箱機能は逆に危険
  - 「捨てるべきデータが見える」状態は法令対応上問題
- ゴミ箱を持たないことが、将来の法令対応で有利に働く

#### 結論

「基本は物理削除、必要な事業所には論理削除を提供する」という方針に落ち着いた。

これは設計判断としても、ビジネス戦略としても優れている：
- 顧客のニーズに応じたカスタマイズ可能性
- 機能追加としての販売ポイント
- 段階的な実装が可能
- 要件定義書がシンプルになる

## 4. 第2のシナリオ：さくらサーバーのアカウント削除ミス

### 確定事項

- **対策方針**
  - バックアップは作成時に暗号化（mysqldump | openssl）
  - さくらサーバー内には一時ファイルとして生成、R2転送後に削除
  - 永続保管はCloudflare R2のみ
  - 復元時はR2から取得

- **暗号化方式**
  - mysqldump | openssl enc -aes-256-cbc -salt -pbkdf2 -kfile {keyfile}
  - パイプ処理で平文SQLをディスクに書かない
  - AES-256-CBC + PBKDF2

- **鍵管理**
  - 専用鍵ファイル（権限600、Git管理外）
  - 1Password保管、年1回ローテーション
  - システムごとのディレクトリで分離

- **R2の構成**
  - 1バケット、機関ごとにプレフィックス分離
  - 例：r2://大内さん-backups/counseling-cloud-機関A/

### この結論に至った経緯

#### さくら標準バックアップの調査結果

さくらの「バックアップ&ステージング」機能を調査した結果、致命的な制約が判明した：

- 「サービスの解約や料金滞納などによりご契約サービスが廃止となった場合は、バックアップは削除されます」
- データベースのスナップショットは、**WordPressを利用している場合のみ利用可能**
- Laravelで構築する今回のシステムでは、**さくらの標準DBバックアップ機能は使えない**

#### 結論：さくらアカウントの外への保管が必須

「ミスでアカウント停止」「ミスで契約解除」のシナリオに備えるには、さくらアカウントの外にバックアップを持つ仕組みを作るしかない。

#### Cloudflare R2の選定理由

| サービス | 月額(7GB) | エグレス料金 | 選定 |
|---|---|---|---|
| **Cloudflare R2** | 約15円 | 無料 | ◎ |
| AWS S3 | 約25円 | $0.09/GB | △ |
| AWS S3 Glacier | 約4円 | 取り出し時に課金 | △ |
| Backblaze B2 | 約7円 | $0.01/GB | ○ |

**Cloudflare R2を選定した理由**：
1. エグレス料金無料（復元時のコストを気にしなくて良い）
2. S3互換API
3. 料金が分かりやすい
4. 10GB/月までは無料枠
5. 4機関程度なら実質無料で運用可能

#### 「さくら + R2の二重保管」から「R2のみ」へ

当初は「さくらサーバー上にもバックアップを置きつつ、R2にも転送」という二重保管案だったが、議論を通じて以下に整理された：

- さくらサーバー上にバックアップを残す合理性が薄い
- 復元時の速度を気にする必要がない（暗号化済みファイルは小さい）
- R2自体が高耐久性
- 二重管理の煩雑さを避ける

W-1案（R2のみに保管、さくらサーバー上には一時ファイルのみ）を採用。

### 場所と役割の整理

**さくらサーバー**
```
/home/y-ouchi-inmylife/
├── www/
│   ├── counseling/         ← システムA本体
│   └── system-b/           ← 将来の別システム
├── secrets/
│   ├── counseling/
│   │   ├── backup.key       ← 暗号化鍵
│   │   └── r2-credentials.txt  ← R2アクセスキー
│   └── system-b/
│       └── ...
└── tmp/                     ← 一時ファイル（転送後削除）
    └── counseling/
        └── backup_*.sql.enc
```

**Cloudflare R2**
```
大内さん-backups/             ← 1つのバケット
├── counseling-cloud-機関A/
├── counseling-cloud-機関B/
├── counseling-jigyo-x/
└── counseling-jigyo-y/
```

**大内さんの手元**
- 通常は何も置かない、緊急時のみR2からダウンロード

### バックアップの流れ

```
時刻 03:00  cron起動
  ↓
さくらサーバー上で:
  1. mysqldump | openssl enc を実行
     → /tmp/counseling/backup_*.sql.enc が生成される
  2. aws s3 cp で R2 に転送
     → R2に保存
  3. /tmp/ のファイルを削除
     → さくらサーバー上にはバックアップが残らない
  4. ログ記録、メール通知
```

### 複数システム稼働を見越した設計

将来、さくらサーバー1アカウントで複数システムが稼働する可能性を考慮：

- **方向P-3（中間案）**を採用：構造は再利用前提だが、深追いはしない
  - ディレクトリ構造、鍵管理、cron設定の方針を「再利用しやすい形」にしておく
  - 設定ファイルから暗号化鍵パス、R2バケット名・プレフィックスを読む
  - 「次のシステムを作る時、今回の構造を真似すれば動く」というレベル

## 5. 第3のシナリオ：ローカルLAN環境でのサーバー破損

### 確定事項

- **システム管理者**：大内さん（クラウド版と同じ）
- **訪問頻度**：月1回（有償メンテナンス）
- **緊急時対応**：緊急訪問（移動コスト許容）
- **機関側の権限**：バックアップ画面なし（K1）
- **リモート接続**：基本的になし

- **環境前提**
  - サーバー機：両事業所ともWindowsクライアントOS（Win10/11）
  - インターネット接続：あり
  - 24時間稼働しない（営業時間外にPCを片付ける）

- **対策方針（クラウド版と統一）**
  - 基本：R2への自動転送（クラウド版と同じ）
  - 例外：クラウド抵抗ケースはローカル保管（機関選択）

- **実行タイミング**
  - 毎週○曜日 12:30（昼休み中）
  - タスクスケジューラの「実行できなかった場合、できるだけ早く起動」をON
  - PCが立ち上がっていない場合、起動時に自動実行される

- **頻度・世代**
  - 週次×7世代保持（クラウド版と統一）

- **月次持ち帰り運用**
  - 廃止（R2への自動転送で十分）
  - 月次訪問は別の目的（メンテナンス、トラブル対応）に集中

- **Windowsクライアント環境の制約**
  - 同時接続数の制約（Win 10/11は約10〜20）
  - Windows Update再起動による停止リスク
  - 電源管理（スリープ・シャットダウン）
  - これらは契約書に明記、機関側責任とする

### この結論に至った経緯

#### 営業時間外問題の発覚

午前の議論終盤で「営業時間外にPCを片付ける」運用が判明し、深夜03:00の定時実行が成立しないことが分かった。

#### お散歩での解決アイデア

「24時間稼働しないなら、稼働時間中にやればいい」という発想転換。

タスクスケジューラの「スケジュールされた時刻にタスクを開始できなかった場合、タスクをできるだけすぐに開始する」設定により：
- 12:30 にPCが起動していなければスキップ
- PCを起動した時点で「実行漏れ」を検知して、自動的に実行される
- 結果として「起動時実行」と「定時実行」を1つの設定で実現

#### LAN版もR2転送に統一

「LAN版でもインターネット接続はある」という情報から、LAN版もクラウド版と同じくR2転送を基本とした。これにより：

- クラウド版とLAN版の設計が統一
- 月次持ち帰り運用が不要に
- 大内さんの月次訪問はメンテナンスに集中できる

### 提供形態の整理（最終版）

- **提供形態は3パターン**
  1. クラウド版 + R2保管（標準）
  2. LAN版 + R2保管（標準）
  3. LAN版 + ローカル保管（クラウド抵抗ケース）

- **営業上の説明**
  - クラウド希望 → クラウド版 + R2保管
  - LAN希望 + クラウドOK → LAN版 + R2保管
  - LAN希望 + クラウド抵抗 → LAN版 + ローカル保管

- **設計上の意味**
  - クラウド版：保管先R2固定、設定切替なし
  - LAN版：保管先を選択可能（インストール時決定）

「クラウド版+ローカル保管」は理論上存在しない（クラウド版を選んだ時点でクラウド利用に同意している）ため、選択肢から除外。

## 6. バックアップ管理画面の廃止

### 確定事項

- **方針**：バックアップ管理画面を完全廃止

- **廃止する機能**
  - 自動バックアップの時刻設定UI
  - 手動バックアップ実行ボタン
  - バックアップ履歴の一覧表示
  - ダウンロード/復元/削除ボタン

- **代替手段**
  - 自動バックアップ：cron / タスクスケジューラで管理
  - 手動バックアップ：システム管理者がSSHで `php artisan backup:run` を実行
  - 状況確認：R2のダッシュボード、メール通知
  - 復元：システム管理者がSSH経由で対応

### 廃止の根拠

1. **K1（機関側に画面なし）が確定済み**：機関の管理カウンセラーには見せない
2. **R2が真の保管場所**：さくらサーバーの画面で表示するファイルが存在しない
3. **管理ツールはR2のダッシュボードで十分**：Cloudflareが提供する管理UIがある
4. **手動操作はSSH/CLIで実行**：大内さんが Artisan コマンドで実行
5. **時刻設定はサーバー側設定**：cron/タスクスケジューラで管理

### 既存実装の扱い

- **削除対象**
  - routes/web.php のバックアップ関連ルート
  - app/Http/Controllers/Backup* 系コントローラ
  - resources/views/data/backup.blade.php
  - バックアップ関連のテスト

- **新規実装**
  - app/Console/Commands/BackupRun.php（Artisanコマンド）
  - 暗号化処理（mysqldump | openssl のラッパー）
  - R2転送処理（AWS SDK等）
  - メール通知処理

- **設定追加**
  - config/backup.php（バックアップ設定）
  - .env.example にR2関連の設定項目
  - 鍵ファイルの配置パス

## 7. 定時実行の方法（環境別）

### さくらサーバー（共有サーバー）の場合

#### 設定方法

コントロールパネル経由が推奨：
```
コントロールパネル
└── スクリプト設定
    └── Cron設定
        └── 「新規追加」で設定
```

設定例（毎週日曜日 03:00）：
```
分: 0
時: 3
日: *
月: *
曜日: 0
コマンド: cd /home/user/www/counseling && /usr/local/bin/php artisan backup:run >> /home/user/backup-cron.log 2>&1
```

#### 注意点

- PHPコマンドの絶対パス指定が必要（`/usr/local/bin/php`）
- 環境変数とパスの問題（必ず `cd` で作業ディレクトリ明示）
- シェルがtcshであるため、bash構文（`2>/dev/null`等）は使えない
- 実行頻度の制限あり（同時実行数等）

### Windowsローカル（タスクスケジューラ）の場合

#### 設定方法

「タスクの作成」を選択（「基本タスクの作成」より高機能）。

**「全般」タブ**
- 名前：`Counseling Backup`
- ユーザーがログオンしているかどうかにかかわらず実行する：ON
- 最上位の特権で実行する：ON

**「トリガー」タブ**
- 設定：毎週
- 開始時刻：12:30:00
- 曜日：実行したい曜日にチェック

**「操作」タブ**
- 操作：プログラムの開始
- プログラム/スクリプト：バッチファイルのパス（推奨）
  - 例：`C:\path\to\counseling\backup.bat`

**「条件」タブ**
- タスクを実行するためにスリープを解除する：ON

**「設定」タブ（重要）**
- **スケジュールされた時刻にタスクを開始できなかった場合、タスクをできるだけすぐに開始する：ON**
- タスクが失敗した場合の再起動の間隔：30分、3回試行
- タスクの実行時間が次の値を超えたら停止する：1時間

#### バッチファイルの例

`C:\path\to\counseling\backup.bat`:
```batch
@echo off
cd /d C:\path\to\counseling
C:\xampp\php\php.exe artisan backup:run >> backup.log 2>&1
```

#### 注意点

- PHP の実行パス確認が必要
- 「ユーザーがログオンしているかどうかにかかわらず実行する」設定が重要
- バッチファイル経由だと運用しやすい
- 「StartWhenAvailable」設定が「実行漏れ補完」の核心

### バックアップスクリプトの中身

定時実行で起動される `php artisan backup:run` の内部処理：

1. 一時ディレクトリの準備
2. mysqldump | openssl で暗号化バックアップ作成
3. R2 にアップロード（AWS SDK等を使用）
4. 古いバックアップの削除（R2側、7世代より古いもの）
5. ログ記録
6. 一時ファイル削除
7. エラー時はメール通知

役割の分担：
```
cron / タスクスケジューラ
↓ 「決まった時刻に起動する」役目だけ
php artisan backup:run
↓ 「実際のバックアップ処理」を行う
```

## 9. 設計判断の主要な原則

議論を通じて確立された原則：

### 「最も操作スキルが低い管理者を基準に設計する」
- 機関にシステム管理できる人がいる/いないが分からないため、操作させない設計を選択
- 業務システムの安全設計の原則

### 「システム管理者は業務データに触れない」
- システム管理者（大内さん）は、クライアントや相談記録の操作画面にアクセスできない
- 個人情報保護の観点で健全な設計

### 「機能として存在しないものは事故を起こせない」
- ダウンロード機能を機関側に出さない（K1）
- バックアップ管理画面を完全廃止
- 構造的にリスクを排除する設計を優先

### 「業務実態に立ち返って設計を検証する」
- お散歩での思考転換が、複雑な設計をシンプルにした
- 「想定」より「実態」を信頼する
- 過剰設計を避ける

### 「業界スタンダードの中で最適化する」
- 完全に物理削除：銀行系・医療系の一部
- 論理削除のまま長期保持：一般的なSaaS
- 一定期間後に自動物理削除：Gmail等
- 今回は「物理削除のまま、業務ルールガード」が業務実態に最も適した選択

### 「契約と技術的設計の責任分界の対応」
- 月次訪問・サポート＝有償サービス
- WindowsクライアントOSの制約は機関側責任
- LAN版+ローカル保管のリスクは契約書に明記

### 「将来の拡張余地を残す」
- 物理削除を基本としつつ、論理削除は将来の拡張機能として用意
- 「データを残したい事業所」が現れたら個別対応
- バックアップ転送の仕組みを汎用化（複数システム対応）
